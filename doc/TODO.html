		[ ]  "Hazardous" cycles, i.e. those that require cycle accurate

		     or device access. Think about this...
		     UNFORTUNATELY also normal read/writes may interfere :(
		     so for now, I only feel comfortable with the following:
		     
		     1) Simple case: only ONE component is executable. Then it
		        can of course be executed chunk-wise however is most
		        convenient.

		     2) Semi-simple case 1:  ONE component is "fast" (say, 100 MHz),
		        and one or more are slow (say, 100 Hz). The difference
		        is 1 M times, so the fast component could run lots of
		        chunk-wise execution passes, with e.g. 10000 cycles per
		        chunk, before the slow ones need to run at all. THEN
		        they are interleaved in a single-cycle manner.

		     3) Semi-simple but hardcoded case 2: TWO components run at
		        fixed high speed (e.g. 100 MHz). _IF_ (and there are LOTS

		        dyntrans and so on... then they could be executed in a
		        xyxyxyxyxyxy interleaved manner by the dyntrans system.
		        But that would be horrible. :-|

		     So to preserve correctness, until I've figured out some
		     better way, only cases 1 and 2 will most likely be supported,
		     unless in "sloppy mode" where accuracy is lost and all
		     devices may run chunk-wise (and also reverse execution is
		     turned off).
